¿De dónde vienen los precios?

Esta decisión afecta la base de datos, el flujo de recepciones y la complejidad del sistema. Elige la opción que mejor refleja cómo opera tu laboratorio.

A

Campo manual en el producto

Cada producto tiene un campo precio_referencia que el admin actualiza cuando quiere. Al crear la solicitud, ese valor se copia automáticamente.

Pros

  • Muy simple de implementar
  • El usuario controla el precio

Contras

  • Se desactualiza si no se mantiene
  • No refleja variaciones por proveedor
B

Último precio pagado (desde recepciones)

Cuando se registra una recepción, el sistema guarda el precio unitario. La solicitud sugiere automáticamente el último precio que se pagó a ese proveedor.

Pros

  • Se actualiza solo con cada compra
  • Refleja la realidad del mercado

Contras

  • Requiere agregar precio a las recepciones
  • Sin historial al inicio
C

Catálogo de precios por proveedor

Tabla separada precios_proveedor: cada proveedor puede tener distintos precios para el mismo producto. El admin mantiene este catálogo y se aplica según el proveedor de la solicitud.

Pros

  • Más preciso si hay varios proveedores por producto
  • Permite cotizaciones

Contras

  • Más complejo de mantener
  • Requiere más migración y pantallas
D

Se escribe al momento de la solicitud

El campo precio está en la solicitud misma. Al armar la lista, la persona que la crea escribe el precio estimado. No hay precio guardado en ningún lado.

Pros

  • Cero mantenimiento de catálogo
  • Máxima flexibilidad

Contras

  • Hay que buscarlo cada vez
  • Sin historial de precios

💡 Mi recomendación: B + A como fallback — precio de la última recepción, editable en la solicitud si no hay historial.
