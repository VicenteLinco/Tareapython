Diagnóstico del Sistema — 12 problemas detectados

Revisé el código de todas las páginas. Aquí está el mapa completo antes de definir opciones de solución.

🔴 Bugs — cosas que están rotas hoy
Bug
Stock: filtros Categoría y Proveedor no aplican /stock
El backend SÍ soporta los parámetros categoria_id y proveedor_id. El bug está en cómo el handler de stock construye la query dinámica — los índices de parámetros ($1, $2...) se pueden estar solapando cuando se combinan con otros filtros activos. Requiere revisar stock.rs líneas 76–90.
Bug
Consumos: buscador muestra productos incorrectos /consumos
Cuando se escriben menos de 3 caracteres (ej. "pr"), el query no lleva filtro q y el backend retorna los 100 primeros productos del sistema — da la ilusión de que el buscador funciona mal. El umbral mínimo de 3 chars no es visible para el usuario.
Bug
Solicitudes: buscador del historial no funciona /solicitudes-compra
El input de búsqueda en la pestaña Historial no tiene value ni onChange — es puramente decorativo. No hay estado ni query param conectado. La lista nunca se filtra.
Bug
Solicitudes: generar PDF falla (tipos no coinciden) /solicitudes-compra
La función exportarSolicitudPDF(detail) espera el tipo SolicitudPdfOptions pero recibe un SolicitudDetalle. Los campos no coinciden directamente — falta mapeo. Además, IVA se calcula hardcodeado al 19% CLP, no usa configuración de moneda.
Bug
Solicitudes: "Sugerir compra" desde dashboard no agrega producto Dashboard → /solicitudes-compra
El botón en el dashboard navega a /solicitudes-compra?select=PRODUCTO_ID, pero la página de solicitudes nunca lee ese parámetro de URL. El producto nunca se pre-agrega al borrador.
Bug
Conteo: crear sesión con áreas sin stock falla o confunde /conteo
Al intentar crear una sesión, si el área no tiene items en stock el backend puede retornar error o la sesión queda vacía. Las áreas vacías aparecen en el modal de selección sin advertencia. El usuario no sabe qué áreas son viables.
🟡 Mejoras UX — funciona pero podría ser mucho mejor
UX
Dashboard: tipografía inconsistente con el resto del sistema Dashboard
Usa font-black muy agresivo, texto de 9–10px en múltiples lugares y una mezcla de estilos que no sigue el patrón del resto de páginas (que son más limpias y usa font-semibold). Comparación visual disponible.
UX
Consumos: sin filtro por área, poca información de producto, sin selección de lote /consumos
No hay filtro de área. Los productos muestran poco contexto (no se ve categoría, área, proveedor). No hay forma de elegir un lote específico — solo FEFO automático. Sistemas como Fracttal y Megastock ofrecen selección de lote en consumos.
UX
Conteo: indicador de urgencia de áreas no es configurable /conteo
Las áreas marcadas como "pendientes" usan el campo conteo_frecuencia_dias por área. El usuario quiere que el período sea configurable globalmente desde Panel de Control, como punto de partida para todas las áreas.
UX
Recepciones: faltan elementos clave en la vista de detalle /recepciones
La vista de detalle tiene estructura básica pero falta: editar items de un borrador, vincular con solicitud de compra, resumen de totales claros, y acciones de flujo (aprobar/rechazar). Necesito que listes qué es lo más importante que falta — tengo varias opciones.
🔵 Funcionalidad nueva
Nuevo
Configuración: moneda configurable (símbolo + código) Panel de Control
Agregar campos moneda_codigo (CLP, USD, PEN, COP...) y moneda_simbolo ($, S/, etc.) en la tabla configuracion. Usar en precios del catálogo, solicitudes de compra y PDF.
Nuevo
PDF Solicitudes: incluir logo del laboratorio /solicitudes-compra → PDF
El logo ya se guarda como base64 en configuracion.logo_base64. Falta: leer esa configuración al generar el PDF y renderizar la imagen en el encabezado. La función exportarSolicitudPDF ya recibe nombreLaboratorio pero no el logo.
